::before cannot anchor to its originating element via explicit anchor-name

Summary
When a ::before pseudo-element is absolutely positioned and references its originating element via an explicit position-anchor: --name, Chrome fails to resolve the anchor: position-area is ignored and the pseudo falls back to the static/top-left position of its containing block. The same configuration with position-anchor: auto (the default, which per spec resolves to the originating element) works correctly.
Spec
[ANCHOR-POSITION-1] §3.1 Acceptable Anchor Element, §3.2 Implicit Anchor Element, §3 Finding an Anchor
Test
css/css-anchor-position/ — see the matrix in the footer.
Env
Chrome 147.x (arm64, macOS)
Repro
Case 1 is broken. Case 2 and Case 3 work. Compare the three boxes below.

Case 1 — explicit anchor-name on originating element (broken)

Topology: .container { anchor-name; position: relative }, .container::before { position-anchor: --anchor-1 }. The anchor element is the containing block of the positioned pseudo.

Case 2 — explicit anchor-name on a descendant of the containing block (works)

Topology: .outer { position: relative }, .target { anchor-name } child, .outer::before { position-anchor: --anchor-2 }. The anchor element is a strict descendant of the containing block.

Case 3 — implicit anchor (position-anchor: auto) on originating element (works)

Topology identical to Case 1, but the pseudo does not specify position-anchor. Per spec, the implicit anchor is the originating element. This path succeeds — so the originating element is reachable as an anchor; only the explicit name lookup fails.

Analysis

1. What the spec says

[ANCHOR-POSITION-1] §3.2:

The implicit anchor element of a pseudo-element is its originating element, unless otherwise specified.

And §3.1 defines the algorithm for resolving an explicit anchor-name. An element possible anchor is an acceptable anchor element for an absolutely positioned element positioned el if all of:

  1. possible anchor is either an element or a fully styleable tree-abiding pseudo-element.
  2. possible anchor is in scope for positioned el, per the effects of anchor-scope.
  3. possible anchor is laid out strictly before positioned el, which holds when one of:
    • they share the same original containing block and the anchor is either non-absolutely-positioned or comes earlier in flat tree order, or
    • the element generating possible anchor's containing block is itself an acceptable anchor element for positioned el (recursive).
  4. Skipped-contents consistency (c-v).

2. Why Case 1 should resolve per spec

For .container::before (the positioned element), the containing block is .container itself (since .container is position: relative). The candidate anchor is also .container. They do not share a containing block, so the recursive branch of rule 3 applies:

  • .container's containing block is its parent (the <section>).
  • That parent is an acceptable anchor element for .container::before (same containing block as .container, laid out earlier in flat tree order), so the recursion terminates successfully.

Therefore .container is an acceptable anchor for its own ::before. Nothing in the spec excludes the case where the anchor equals the containing block of the positioned element.

3. What Chrome actually does

Three observations from the cases above pin down the failure:

  • Case 3 works → the originating element is reachable as an anchor at layout time; there is no fundamental containing-block-is-anchor problem.
  • Case 2 works → explicit anchor-name resolution works when the anchor is a strict descendant of the containing block.
  • Case 1 fails → only the combination "explicit anchor-name" + "anchor equals containing block" fails.

The symptom — position-area silently ignored, pseudo rendered at the top-left of its containing block — matches the behavior of a positioned element whose position-anchor lookup returned "no acceptable anchor", in which case anchor-dependent properties are treated as their initial values.

The likely implementation-level cause: the anchor-name lookup walks the flat tree starting from the positioned element, searching for an element with the matching name that is also an acceptable anchor. The check "is this anchor strictly laid out before the positioned element" is likely implemented as "the anchor must be an element that finishes layout before the positioned element begins layout". When the anchor is the containing block, its box layout is still in progress at the moment the descendant absolutely-positioned box is laid out, so the check returns false. The implicit-anchor path in Case 3 bypasses the name-based search and takes a different code path that knows the originating element is already laid out enough to supply geometry.

4. Ruling out size containment

The directory name for this repro originally hinted at size containment (contain: size / container-type: size) as the suspected cause. After instrumenting the cases, size containment is not involved: no case uses contain, container-type, or any containment-triggering property. The triggering condition is strictly topological (anchor = containing block) combined with the explicit-name resolution path.

6. Real-world impact

This combination is common in decorative patterns:

  • A card component with anchor-name declared on the card root for use by external tooltips, combined with a ::before badge on the same card positioned via position-area.
  • Authors who name their anchor explicitly (for documentation/readability) expect it to behave identically to the implicit default. The silent divergence is a footgun: removing the redundant-looking position-anchor: --name line "fixes" the layout with no diagnostic.

Workaround: omit position-anchor on pseudo-elements whose anchor is their originating element, relying on the implicit default. Or move the anchor-name onto a dedicated child element (Case 2 topology).
